## Context

当前应用使用 React 19 + TypeScript + Tailwind CSS 构建，主要显示鸟类声音识别结果。结果页面包含两个主要部分：
1. 实时识别结果卡片列表（显示在 `src/App.tsx` 的 `renderResultsList` 函数中）
2. 历史记录列表（显示在 `src/App.tsx` 的 `renderHistoryList` 函数中）

当前问题：
- 在窄宽度移动设备上，结果卡片的内容可能溢出容器
- 字体大小和间距在移动端不够优化
- 历史记录条目在移动端显示不完整
- 缺乏系统的响应式设计策略

## Goals / Non-Goals

**Goals:**
1. 确保结果卡片在移动端（最小宽度 320px）上完全可见，无内容溢出
2. 优化字体大小、行高和间距，提升移动端可读性
3. 实现响应式布局，自动适应不同屏幕尺寸
4. 保持现有桌面端体验不变或更好
5. 使用 Tailwind CSS 的响应式工具类，避免自定义媒体查询

**Non-Goals:**
1. 不重写整个应用架构
2. 不改变现有功能逻辑
3. 不引入新的 JavaScript 框架或库
4. 不修改后端 API 或数据模型

## Decisions

1. **响应式策略**: 使用 Tailwind CSS 的移动优先响应式工具类（如 `sm:`, `md:`, `lg:`）
   - 理由: 项目已使用 Tailwind CSS，这是最符合现有技术栈的解决方案
   - 替代方案: 自定义 CSS 媒体查询（会增加维护成本）

2. **布局调整**:
   - 结果卡片: 在移动端使用更紧凑的布局，减少左右 padding，调整图片尺寸
   - 历史记录: 在移动端使用单列布局，优化信息密度
   - 理由: 移动端屏幕空间有限，需要优化信息展示

3. **字体大小调整**:
   - 使用 Tailwind 的响应式文本类（如 `text-sm sm:text-base`）
   - 理由: 确保在不同屏幕尺寸下的可读性

4. **断点选择**:
   - 移动端: `< 640px` (Tailwind 的 `sm` 断点以下)
   - 平板端: `640px - 1024px`
   - 桌面端: `≥ 1024px`
   - 理由: 使用 Tailwind 的标准断点，保持一致性

## Risks / Trade-offs

1. **风险**: 过度调整可能影响桌面端体验
   - **缓解**: 使用 Tailwind 的响应式类，确保桌面端样式不变

2. **风险**: 不同设备测试覆盖不全
   - **缓解**: 使用 Chrome DevTools 的设备模拟器测试常见设备尺寸

3. **权衡**: 移动端紧凑布局 vs 信息完整性
   - **决策**: 优先保证关键信息可见，必要时使用省略号或折叠次要信息

4. **风险**: 现有 CSS 类可能冲突
   - **缓解**: 仔细检查现有样式，使用特异性更高的选择器或修改现有类
